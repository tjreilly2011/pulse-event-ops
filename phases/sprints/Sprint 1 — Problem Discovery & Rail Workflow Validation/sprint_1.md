# Sprint 1 — Problem Discovery & MVP Workflow Definition

## Phase
PHASE 0 — Discovery & Product Definition

## Goal
Identify and validate a single high-value operational workflow to solve, based on real frontline pain, and define the simplest possible MVP use case to build next.

---

## Why This Sprint Matters
This product only succeeds if it solves a real, repeated problem experienced by frontline staff.

This sprint is about grounding the product in reality before any architecture or implementation decisions are made.

No code exists yet — this is a discovery and definition sprint.

---

## Wishlist

- Identify the primary frontline user (rail CSO / guard)
- Identify supporting roles:
  - station staff
  - management/control
- Capture 3 real operational scenarios where communication breaks down
- Describe how those scenarios are currently handled (calls, WhatsApp, etc.)
- Identify the biggest friction points (delays, lack of info, missed coordination)

- Select ONE scenario to focus on (highest value, most frequent, easiest to improve)

- Describe that scenario step-by-step:
  - what triggers it
  - who is involved
  - what actions are taken
  - what goes wrong

- Define the desired outcome:
  - what “good” looks like
  - what should happen instead

- Define the MVP workflow:
  - who creates the event
  - who receives it
  - what action is expected
  - what acknowledgment/update looks like

---

## Constraints

- Focus on ONE workflow only
- Do not design full system
- Do not design database or APIs
- Do not generalize across industries yet
- Do not introduce technical solutions prematurely

This is a reality-first sprint, not a design sprint.

---

## Out of Scope

- Detailed architecture
- Data model or schema design
- Realtime implementation details
- Mobile or dashboard UI design
- Multi-vertical abstraction
- Messaging systems

---

## Success Criteria

- One clear operational scenario is selected
- Current broken workflow is clearly described
- Target users and roles are identified
- A simple “before vs after” is defined
- A single MVP workflow is agreed and documented

---

## Deliverables

- Problem statement (clear and specific)
- User roles summary
- “Current state” workflow (what happens today)
- “Desired state” workflow (what should happen)
- MVP workflow definition (simple, actionable)

---

## Notes

This sprint intentionally avoids technical design.

The output of this sprint will feed directly into Sprint 2, where:
- domain model
- event structure
- lifecycle
will be defined based on a real, validated workflow.
